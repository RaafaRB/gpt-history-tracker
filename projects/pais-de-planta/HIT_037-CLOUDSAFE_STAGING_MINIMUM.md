# HIT_039 - Cloud-safe backend minimum for Cloud Run staging (2026-03-19)

This HIT implemented the minimum backend changes required for a reliable serverless staging deployment on Cloud Run using Supabase Postgres, Supabase Storage, and Cloud Scheduler. Database wiring was updated to support Postgres via DATABASE_URL (Supabase-friendly URL normalization and SSL expectation) while preserving SQLite as a local fallback. The Postgres driver dependency was added and smoke-tested in both local and staging-simulated configurations.

Uploads were refactored behind a StorageService abstraction that selects between local disk and Supabase Storage using STORAGE_PROVIDER. Upload endpoints now persist a stable storage key pattern rather than absolute filesystem paths. A lightweight proxy endpoint (GET /storage/{key}) returns a 302 redirect either to local static assets or to a fully qualified short-lived signed Supabase URL, keeping the frontend compatible without UI changes and avoiding container-disk persistence.

Embedded APScheduler behavior was replaced with a protected POST /scheduler/trigger endpoint controlled by SCHEDULER_TRIGGER_TOKEN, with ENABLE_SCHEDULER=false disabling background scheduling in Cloud Run. Concurrency was hardened for multi-instance execution using Postgres row-level locking (FOR UPDATE SKIP LOCKED) plus an atomic status transition to a new processing state to prevent cross-instance double-send even under retries. End-to-end smokes passed in dev mode and staging-simulated mode with correct credentials, and failure behavior with invalid Supabase auth was safe with no secret leakage.
